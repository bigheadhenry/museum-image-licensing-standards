# 图片授权对外说明、内部流程与执行文书三层对应关系（研究稿）

## 一、本轮目标
在上一轮已经形成《图片授权对外说明体系与内部办理链对应关系（研究稿）》的基础上，本稿继续向制度落地方向推进一步：

- 对外说明文本负责向外部用户解释什么；
- 内部流程负责馆内实际如何办理；
- 执行文书负责每个节点如何被记录、确认、通知、控制和留痕。

本稿的重点，是把现有成果从“双层结构”进一步推进为“三层结构”：

**对外说明层 → 内部流程层 → 执行文书层**

这样可以让 `mils` 从“制度研究 + 页面说明 + 流程说明”进一步进入更接近真实机构运行的制度原型包阶段。

## 二、基本判断：制度落地不能只靠页面和流程，还必须有文书承接
如果只有对外说明文本和内部流程，而缺乏执行文书，实际运行中容易出现以下问题：
1. 页面上说得很清楚，但承办时没有稳定表单承接；
2. 流程里写了补正、审核、交付、整改、复核，但实际办事时没有书面节点固定结果；
3. 不同承办人对同类事项处理口径不一，因为缺少统一模板；
4. 事后争议出现时，难以证明当时到底要求了什么、确认了什么、交付了什么；
5. 培训、统计、复盘和制度优化时，缺乏标准化材料来源。

因此，本稿提出一个进一步判断：

> 对外说明负责“让外部理解”，内部流程负责“让馆内运转”，执行文书负责“让每一个节点可被固定、追溯与复核”。

三者必须同时存在，才能真正支撑制度实施。

## 三、建议理解的三层结构
### （一）对外说明层
作用是：
- 向申请人、使用者、平台访问者、开发者和公众解释规则、路径、边界与风险；
- 降低误解，提前分流；
- 把馆内逻辑转译为外部可理解语言。

### （二）内部流程层
作用是：
- 明确馆内在受理、补正、审核、审批、收费、交付、监督、整改、复核等环节具体怎么运转；
- 将事项从“个案经验”变成“稳定链条”。

### （三）执行文书层
作用是：
- 将每个关键节点转化为可执行、可签署、可确认、可留档、可追责的书面载体；
- 让内部流程不只是“原则上应该这样做”，而是“确实通过一张表、一份单、一份通知、一份授权文本被固定下来”。

## 四、建议形成的“三层对应”主结构
### （一）基础公开规则层 → 权利边界识别与平台入口管理 → 统一规则文本与标准标签表达
**对外说明层**：
1. 《版权声明与网站使用规则》；
2. 《官网平台与数字资源公开使用边界说明》。

**内部流程层**：
- 权利边界识别；
- 平台入口管理；
- 公开展示与正式授权边界划分；
- 平台、栏目、资源级信息架构配置。

**执行文书层 / 文本承接**：
- 《版权声明与网站使用规则》本身；
- 《官网平台与数字资源公开使用边界说明》本身；
- 《标准署名与声明示例》；
- 《资源详情页权利标签说明》作为单项资源状态表达工具。

这一层的特点是：
- 文书承接本身更多体现为“公开规则文本”和“标签表达文本”，而不是个案审批单据。

### （二）开放资源分流层 → 资源分级与开放规则判断 → 标签、规则页与标准提示语
**对外说明层**：
1. 《开放资源使用说明》；
2. 《资源详情页权利标签说明》。

**内部流程层**：
- 资源开放等级判断；
- 标签配置；
- 开放资源与需申请资源分流；
- 附条件开放规则设置。

**执行文书层 / 文本承接**：
- 《开放资源使用说明》；
- 《资源详情页权利标签说明》；
- 《标准署名与声明示例》；
- 必要时通过资源详情页标签字段、下载提示语和栏目提示语共同固定使用边界。

这一层的特点是：
- 不一定形成个案申请文书，但必须有“状态文本 + 标签文本 + 标准提示语”共同承接规则。

### （三）页面提示与动作边界层 → 前置风控节点 → 动作提示语与标准确认文本
**对外说明层**：
1. 《数字资源栏目使用提示》；
2. 《下载页面提示语》；
3. 《批量下载与再传播提示》；
4. 《开放 API 与数据接口使用提示》；
5. 《开发者注册与接口申请说明》。

**内部流程层**：
- 页面动作风险控制；
- 下载前提示；
- 批量获取前提示；
- API 接口访问边界控制；
- 开发者注册与申请分流。

**执行文书层 / 文本承接**：
- 栏目级提示语；
- 下载弹窗或确认语；
- API 使用提示文本；
- 开发者注册与接口申请说明页；
- 必要时配合开发者申请表单、接口申请记录或技术协议（现阶段可先以说明文本为主）。

这一层的特点是：
- 主要通过“页面动作前的提示与确认文本”承接风控，而不是传统纸质文书。

### （四）正式申请入口层 → 申请入口管理与形式受理 → 申请表
**对外说明层**：
1. 《图片授权申请说明页》。

**内部流程层**：
- 申请入口管理；
- 申请字段设置；
- 形式受理；
- 最低受理门槛判断。

**执行文书层 / 文书承接**：
1. 《图片使用申请表》。

这是最清晰的一组对应关系：
- 对外说明页告诉申请人“要准备什么”；
- 内部受理环节判断“是否具备进入流程的最低条件”；
- 《图片使用申请表》把主体、用途、图片对象、传播范围、修改、减免、承诺等关键事实正式收进来。

### （五）办理分流解释层 → 补正机制、审核分流与审批边界控制 → 补正通知、审核意见、决定类文书
**对外说明层**：
1. 《图片授权申请补充材料与补正说明》；
2. 《授权申请被暂缓、限制或不予同意情形说明》。

**内部流程层**：
- 补正机制；
- 实质审核；
- 简易/标准/强化审查分流；
- 附条件同意、暂缓决定、不予同意等审批边界控制。

**执行文书层 / 文书承接**：
1. 补正通知（当前项目尚可继续补一份正式《补正通知单》示范稿）；
2. 《审核意见单》；
3. 《收费通知与减免确认单》；
4. 对暂缓、限制、不予同意事项，后续可继续补入更明确的《暂缓决定告知单》或《不予同意通知单》示范稿（现阶段先由说明文本 + 审核/审批结果承接）。

这一层说明：
- 对外页面只能解释“为什么可能会这样”；
- 真正把“会不会这样、依据是什么、条件是什么”固定下来的，是审核意见与决定类文书。

### （六）服务办理层 → 文件规格判断、收费确认、授权书生成与交付前核验 → 收费单、授权书、交付单
**对外说明层**：
1. 《图片服务办理说明》；
2. 《高规格文件与专项服务说明》。

**内部流程层**：
- 文件规格判断；
- 技术/服务支持判断；
- 收费或减免确认；
- 授权条件书面化；
- 交付前核验。

**执行文书层 / 文书承接**：
1. 《收费通知与减免确认单》；
2. 《图片授权书》；
3. 《图片交付记录单》。

这一层是制度执行中最典型的“文书链”：
- 先确认费用与减免；
- 再出授权书固定权利边界；
- 最后通过交付记录单把文件版本、接收对象和交付条件留痕。

### （七）FAQ / 高频解释层 → 统一口径管理、培训与制度优化 → 培训要点、案例演练、统计与优化材料
**对外说明层**：
1. 《常见问题说明》。

**内部流程层**：
- 高频误解归纳；
- 统一答复口径管理；
- 培训回流；
- 台账统计与制度优化。

**执行文书层 / 文书承接**：
1. 《图片授权承办与审核培训要点》；
2. 《图片授权典型案例与办理演练》；
3. 《授权办理留档、统计与制度优化工作说明》；
4. 必要时由 FAQ 修订记录、培训材料、统计分析报告共同承接。

这一层说明：
- FAQ 看似是面向外部的问答页；
- 但它的内部文书基础其实是培训、案例、统计与制度优化材料。

### （八）授权后监督与违规处置层 → 巡查、整改、升级处置、名单管理与复核 → 整改单、撤销单、名单表、复核单、限制决定单
这一层虽然上一轮更多是在“内部办理链”中展开，但实际上也与对外说明直接相连，尤其是：
- 《对外声明与严正声明》；
- 《授权申请被暂缓、限制或不予同意情形说明》；
- FAQ 中关于违规后果的解释。

**对外说明层**：
1. 《对外声明与严正声明》；
2. 《常见问题说明》中关于违规后果的部分；
3. 《授权申请被暂缓、限制或不予同意情形说明》中关于后续限制的部分。

**内部流程层**：
- 授权后监督；
- 事实核实；
- 违规分级判断；
- 顺序升级 / 并行升级 / 直接跳级处置；
- 风险名单管理；
- 后续申请限制；
- 复核、调整与解除。

**执行文书层 / 文书承接**：
1. 《整改通知单》；
2. 《授权撤销或终止通知单》；
3. 《风险名单记录表》；
4. 《风险名单复核意见单》；
5. 《暂停受理或限制后续申请决定单》；
6. 《后续申请强化审查意见单》；
7. 《对外声明与严正声明》。

这一层最能体现“流程如果没有文书就很难成立”：
- 因为整改、撤销、名单管理、限制申请、解除关注，全部都必须通过正式文书才能稳定执行和后续复核。

## 五、建议理解的几条典型“三层链路”
### 1. 申请进入链
- **对外说明**：`图片授权申请说明页`
- **内部流程**：申请入口管理 → 形式受理
- **执行文书**：`图片使用申请表`

这条链说明：
- 申请说明页负责让申请人知道“怎么进门”；
- 申请表负责把“进门”这件事正式固定下来。

### 2. 补正与审核链
- **对外说明**：`图片授权申请补充材料与补正说明`
- **内部流程**：补正机制 → 实质审核 → 审查路径分流
- **执行文书**：补正通知（建议后续补写） + `审核意见单`

这条链说明：
- 补正说明页是解释；
- 审核意见单才是把判断与理由固定下来。

### 3. 收费、授权与交付链
- **对外说明**：`图片服务办理说明` / `高规格文件与专项服务说明`
- **内部流程**：收费确认 → 授权书生成 → 交付前核验 → 文件交付
- **执行文书**：`收费通知与减免确认单` → `图片授权书` → `图片交付记录单`

这是一条最完整、最典型的三层链路。

### 4. 违规处置链
- **对外说明**：`对外声明与严正声明` / FAQ 中违规后果说明
- **内部流程**：监督发现 → 分级判断 → 处置升级 → 名单管理 → 复核解除
- **执行文书**：`整改通知单` → `授权撤销或终止通知单` → `风险名单记录表` → `暂停受理或限制后续申请决定单` → `风险名单复核意见单`

这条链说明：
- 对外公开表达与内部文书链必须同步，否则外部看到的就只是零散结果。

## 六、当前项目中已基本成型的三类文书群
### （一）前端办理文书群
包括：
- 《图片使用申请表》；
- 《审核意见单》；
- 《收费通知与减免确认单》；
- 《图片授权书》；
- 《图片交付记录单》。

这一组已经基本构成：
- 申请 → 审核 → 收费 → 授权 → 交付

的完整文书闭环。

### （二）授权后处置文书群
包括：
- 《整改通知单》；
- 《授权撤销或终止通知单》；
- 《风险名单记录表》；
- 《风险名单复核意见单》；
- 《暂停受理或限制后续申请决定单》；
- 《后续申请强化审查意见单》。

这一组已经基本构成：
- 问题发现 → 处置升级 → 持续控制 → 后续申请管理 → 复核解除

的完整文书闭环。

### （三）口径、培训与优化支撑文书群
包括：
- 《图片授权承办与审核培训要点》；
- 《图片授权典型案例与办理演练》；
- 《授权办理留档、统计与制度优化工作说明》。

这一组承担的是：
- 把前两组文书沉淀成培训、统计、复盘和制度更新能力。

## 七、当前仍可继续补强的文书缺口
从三层结构角度看，当前项目虽然已相当完整，但仍有几个自然缺口，未来可继续补：
1. **《补正通知单（示范稿）》**：把补正机制从说明层进一步落到正式通知文书；
2. **《暂缓决定告知单（示范稿）》**：把“暂缓决定”从解释层落到正式结果文书；
3. **《不予同意通知单（示范稿）》**：把“不同意”结果写成更明确的正式通知文本；
4. **《开放资源标签配置与后台管理说明（研究稿）》**：把开放资源标签体系从页面层再往管理配置层推进一步；
5. **《FAQ 更新、培训回流与统计修订机制（研究稿）》**：把 FAQ 的维护机制单独固化。

这说明项目下一步已经不再缺“基本制度框架”，而是在补最后几类更细的运行文书和维护机制。

## 八、对制度原型的推进意义
本稿的推进意义主要体现在五点：
1. 让 `mils` 从“对外页面 + 内部流程”的双层原型，进一步升级为“对外页面 + 内部流程 + 执行文书”的三层原型；
2. 让已有模板文书不再只是附件，而明确成为每个流程节点的正式承接工具；
3. 让未来馆方落地时能够更清楚地知道：每个页面背后对应什么流程、每个流程节点又该用哪份文书固定；
4. 让培训、统计、留档和制度优化不再漂浮在制度之外，而是有明确文书来源；
5. 让 `mils` 更接近“可运行制度包”，而不是停留在研究稿集合。

## 九、阶段性结论
博物馆图片授权制度如果要真正落地，至少需要同时具备三层：
1. **对外说明层**：让申请人和公众理解规则；
2. **内部流程层**：让馆内知道事情怎样办理；
3. **执行文书层**：让每一个关键节点被正式固定、通知、确认和留痕。

可将当前项目的三层逻辑概括为：
- 对外说明负责解释；
- 内部流程负责运转；
- 执行文书负责固定。

只有当这三层同时存在并彼此对应时，博物馆图片授权制度才真正从“文本集合”进入“可执行制度原型包”的状态。
