# 审批结果类型与后续文书衔接说明（研究稿）

> 说明：本研究稿用于把图片授权事项在审批阶段可能形成的不同结果类型，与其后续应衔接的文书、办理动作、交付控制和监督留痕要求系统整理成一套更可执行的结构说明。其目标不是重复《授权申请受理、审核与审批工作流程说明（示范稿）》的顺序描述，而是进一步回答：**同意、附条件同意、暂缓决定、不予同意等不同审批结果，分别会把事项带向哪一组后续文书链。**

## 一、为什么需要单独讨论“审批结果类型—后续文书衔接”
在图片授权办理中，审批结果并不是孤立的结论，而是后续办理链条的分流节点。若仅写“审批结果包括同意、附条件同意、暂缓决定和不同意”，但不进一步说明每一种结果后面应接哪些文书、哪些动作、哪些限制和哪些留痕，就容易出现以下问题：
1. 审批结果停留在口头或内部状态，外部申请人难以理解其法律与程序含义；
2. 馆内虽然完成审批判断，但后续收费确认、授权书签发、文件交付和成果反馈之间衔接松散；
3. “附条件同意”与“同意后附提醒”混淆，“暂缓决定”与“继续补正”混淆，“不予同意”与“未继续办理”混淆；
4. 事后争议处理中，难以还原一项申请究竟是在哪个节点被分流、被限制或被终止；
5. 现有模板虽逐步丰富，但缺少一份把它们按审批结果重新组织的总说明。

因此，有必要把审批结果本身视为一个“文书链分流器”：不同结果不仅意味着不同判断，也意味着不同的后续文本、不同的交付可能性、不同的监督需求和不同的再申请条件。

## 二、审批结果不是终点，而是后续文书链入口
从办理结构看，审批结果至少承担四重功能：
1. **确定事项状态**：决定事项当前是可继续、待控制、待搁置还是终止；
2. **触发后续文书**：不同结果会触发不同的通知、确认、授权、交付或监督文书；
3. **限定后续动作**：并非所有审批结果都能进入收费、授权书、交付等环节；
4. **为留档和复核提供依据**：后续是否可重新申请、是否进入强化审查、是否需继续跟踪，都与审批结果类型密切相关。

因此，审批结果不宜理解为“审批结束语”，而应理解为“后续办理链条的结构性入口”。

## 三、四类审批结果及其基本定位
现阶段制度原型中，审批结果主要可分为四类：
1. 同意；
2. 附条件同意；
3. 暂缓决定；
4. 不予同意。

这四类结果的区别，不仅在于“是否允许”，更在于其对后续文书链的开放程度不同：
- **同意**：后续链条以收费确认、授权书、交付和反馈为主；
- **附条件同意**：后续链条以结果告知、收费确认、授权书强化条件、交付前核验和反馈监督为主；
- **暂缓决定**：后续链条以告知、补充说明、方案调整、进一步评估和恢复判断为主；
- **不予同意**：后续链条以结果通知、停止误解、重新申请条件说明和必要时的风险控制为主。

## 四、不同审批结果对应的后续文书衔接结构

### （一）同意路径：进入“确认—授权—交付—反馈”链
“同意”是最接近完整放行的审批结果，但即使如此，也不应被理解为“审批通过 = 可立即拿文件 = 可立即使用”。

#### 1. 推荐衔接文书
“同意”结果后，原则上应衔接以下文书或动作：
1. 《收费通知与减免确认单》；
2. 《图片授权书》；
3. 《图片交付记录单》；
4. 《成果反馈表》；
5. 如涉及公开表达，还可能衔接《标准署名与声明示例》。 

#### 2. 核心逻辑
- “同意”解决的是**是否允许**；
- 《收费通知与减免确认单》解决的是**费用结构是否成立、是否减免、何时履行**；
- 《图片授权书》解决的是**核准用途、期限、媒介、修改权限和限制条件的正式文本化**；
- 《图片交付记录单》解决的是**文件何时、交给谁、交付什么规格、是否完成条件核验**；
- 《成果反馈表》解决的是**使用结果能否回流留档并支持后续监督**。

#### 3. 实务判断
因此，“同意”后通常不再需要《补正通知单》《暂缓决定告知单》《不予同意通知单》或《附条件同意通知单》，而应尽快进入可执行授权链。

### （二）附条件同意路径：进入“结果告知—条件确认—收费/授权—交付核验—反馈监督”链
“附条件同意”不是“同意”的弱化版，而是一个独立结果类型。它意味着馆方原则上允许使用，但只有在附加条件被明确、被接受、被履行后，事项才能继续向收费、授权和交付推进。

#### 1. 推荐衔接文书
“附条件同意”结果后，原则上应衔接：
1. 《附条件同意通知单》；
2. 《收费通知与减免确认单》；
3. 《图片授权书》；
4. 《图片交付记录单》；
5. 《成果反馈表》；
6. 必要时衔接《标准署名与声明示例》或样稿/页面确认材料。

#### 2. 与“同意”路径的区别
附条件同意与同意路径的核心差别在于：
- 其一，必须先通过《附条件同意通知单》把“为什么原则同意、为什么仍需控制、控制条件是什么”正式告知申请人；
- 其二，在进入收费确认、授权书和交付前，应优先确认附加条件已被申请人接受或已完成必要前置条件；
- 其三，后续《图片授权书》往往需要更细地把条件写实，而不只是照常授权；
- 其四，交付前核验和授权后监督的重要性显著高于普通同意事项。

#### 3. 典型条件链
附条件同意路径中，最常见的条件链包括：
- 平台范围限制；
- 文件规格限制；
- 修改与再制作限制；
- 封面/主视觉/广告用途限制；
- 署名与非背书声明要求；
- 成果反馈与后续复核要求。

因此，附条件同意路径不是简单“加一句提醒”，而应形成一条更厚的条件文书链。

### （三）暂缓决定路径：进入“告知—补充/调整—进一步判断”链
“暂缓决定”并不等于不同意，也不当然意味着将来一定同意。它的核心功能，是在当前事实、材料、风险评估或内部确认条件不足时，为事项保留继续判断空间。

#### 1. 推荐衔接文书
“暂缓决定”结果后，原则上应衔接：
1. 《暂缓决定告知单》；
2. 视情况再衔接《补正通知单》；
3. 必要时衔接补充样稿、合作说明、传播方案、专项承诺等辅助材料；
4. 事项恢复判断后，再进入《附条件同意通知单》《不予同意通知单》或同意路径的后续文书链。

#### 2. 核心逻辑
暂缓决定最重要的不是“卡住申请”，而是明确：
- 当前不能直接进入收费确认；
- 当前不能直接签发授权书；
- 当前不能进入文件交付；
- 当前需要先补充、核实、缩小范围、调整方案或等待内部/外部条件明确。

#### 3. 与补正的区别
- **补正**更偏向受理阶段材料不足；
- **暂缓决定**更偏向已经进入审核或审批判断后，仍不足以形成最终结论。

因此，暂缓决定路径可以继续调用补正文书，但其结构位置高于单纯补正。

### （四）不予同意路径：进入“结果通知—边界说明—必要控制”链
“不予同意”是当前申请事项的正式否定结果，但它不必然意味着对申请主体永远关闭后续申请入口。其重点是：用正式文书将当前事项的否定判断、理由和后续边界固定下来。

#### 1. 推荐衔接文书
“不予同意”结果后，原则上应衔接：
1. 《不予同意通知单》；
2. 视情在对外说明层与《授权申请被暂缓、限制或不予同意情形说明》形成口径对应；
3. 如已发生未授权使用、虚假申报、拒不整改或高风险外部传播，还可能继续衔接《整改通知单》《授权撤销或终止通知单》《对外声明与严正声明》《风险名单记录表》或《暂停受理或限制后续申请决定单》。

#### 2. 核心逻辑
不予同意路径的关键不是“停止一切文书”，而是：
- 明确本次申请不能进入收费确认；
- 不能进入授权书签发；
- 不能进入交付；
- 但应保留对重新申请、风险控制和必要处置的制度化承接。

#### 3. 与终止全部关系的区别
不予同意是针对当前申请事项所作出的结论，不当然等于：
- 今后任何事项都不再受理；
- 一定进入风险名单；
- 一定需要公开声明。

是否进一步升级处理，仍需结合有无违规、是否先行使用、是否虚假申报及公共传播风险来判断。

## 五、审批结果与后续动作之间的“允许 / 禁止”边界
从办理控制角度看，不同审批结果对应的后续动作开放程度可以概括如下：

### 1. 可以进入收费确认的结果
- 同意；
- 附条件同意。

### 2. 原则上不得直接进入收费确认的结果
- 暂缓决定；
- 不予同意。

### 3. 可以进入授权书签发的结果
- 同意；
- 附条件同意（且已完成必要条件确认）。

### 4. 原则上不得进入授权书签发的结果
- 暂缓决定；
- 不予同意。

### 5. 可以进入文件交付的结果
- 同意（并完成收费/减免确认及授权书生成）；
- 附条件同意（并完成附加条件确认、收费/减免确认及授权书生成）。

### 6. 原则上不得进入文件交付的结果
- 暂缓决定；
- 不予同意。

### 7. 可以进入成果反馈与后续监督的结果
- 同意；
- 附条件同意；
- 以及已实际发生未授权使用、整改或声明处置事项的不予同意路径。

可见，审批结果本身已经决定了后续办理动作的开放边界。

## 六、审批结果与留档结构的区别
不同审批结果，不仅带来不同文书链，也意味着留档重点不同：

### 1. 同意 / 附条件同意事项的留档重点
- 审核意见；
- 审批结果；
- 收费/减免确认；
- 授权书；
- 交付记录；
- 成果反馈；
- 后续监督记录。

### 2. 暂缓决定事项的留档重点
- 暂缓原因；
- 待补事项；
- 方案调整记录；
- 后续恢复判断节点；
- 最终转入何种结果路径。

### 3. 不予同意事项的留档重点
- 不予同意理由；
- 申请人后续沟通情况；
- 是否存在重新申请可能；
- 是否引发后续整改、声明、名单或限制处理。

换言之，审批结果不只决定“接什么文书”，也决定“以后需要留下什么证据”。

## 七、从“审批结果”看现有文书体系的结构分层
若按审批结果类型重新整理现有文书体系，可大致分为：

### 1. 前置准备与受理文书
- 《图片使用申请表》
- 《补正通知单》

### 2. 审核支撑文书
- 《审核意见单》

### 3. 审批结果分流文书
- 《暂缓决定告知单》
- 《不予同意通知单》
- 《附条件同意通知单》
- （未来如需要，还可补一份更简洁的“同意通知”或由授权书直接承接同意结果）

### 4. 结果后的执行文书
- 《收费通知与减免确认单》
- 《图片授权书》
- 《图片交付记录单》
- 《成果反馈表》

### 5. 授权后监督与升级处置文书
- 《整改通知单》
- 《授权撤销或终止通知单》
- 《风险名单记录表》
- 《风险名单复核意见单》
- 《后续申请强化审查意见单》
- 《暂停受理或限制后续申请决定单》

这说明现有项目已不仅仅是“模板堆积”，而逐步形成了以审批结果为中心的文书链结构。

## 八、阶段性结论
现阶段制度原型的一个重要进展，是审批结果不再只是流程中的抽象节点，而开始成为实际文书链、动作链和留档链的分流中心。

可以形成以下阶段性判断：
1. **同意**应主要导向收费确认、授权书、交付和成果反馈；
2. **附条件同意**应主要导向结果告知、条件确认、收费确认、强化授权文本、交付前核验和更强的后续监督；
3. **暂缓决定**应主要导向补充说明、方案调整、进一步核实和恢复判断；
4. **不予同意**应主要导向结果固定、边界说明、必要时的后续风险控制与再申请管理。

换言之，审批结果的真正价值，不只是告诉申请人“能不能用”，而是决定一项申请接下来进入哪一条制度化文书链。
